# Claude Code 使用技巧深度整理（基于 Advent of Claude 2025）

来源文章：[Advent of Claude 2025](https://adocomplete.com/advent-of-claude-2025/)

整理目标：把文章中的 31 个 Claude Code 技巧，连同每段文字与每个嵌入视频所表达的演示意图，系统化整理成一份可直接上手的中文笔记。

说明：文章中的视频是静音、循环的短演示片段。本文对“视频内容”的理解，基于视频对应的小节主题、正文描述、命令示例和演示画面可确认的信息来做拆解，因此重点放在“作者想通过视频展示的操作路径”和“这些功能在实际工作中的用法”。

---

## 一、这篇文章真正想传达的核心

这篇文章表面上是在列 31 个功能点，底层其实是在讲一套“如何把 Claude Code 用成高杠杆开发环境”的方法论：

1. 先让 Claude 理解项目，再让它开始做事。
2. 用快捷操作降低交互摩擦，让 AI 成为终端工作流的一部分，而不是额外负担。
3. 用计划、权限、回滚、钩子这些机制，保留人的控制权。
4. 把高频提示词、上下文、规则、自动化沉淀下来，让一次性经验变成长期资产。
5. 当你不再只把 Claude Code 当聊天窗口，而是当“可持续运行的开发环境”，它的价值才会真正放大。

作者反复强调的不是“让 AI 替你做完一切”，而是“让 AI 在你设定的轨道里更快、更稳、更可复用地工作”。

## 二、按章节拆解：31 个技巧的深度理解

## 1. Getting Started：先做项目入场配置

### 1. `/init`：让 Claude 自己完成项目入职

视频演示意图：
- 展示在一个现有代码仓库里运行 `/init` 后，Claude 会扫描代码结构并产出 `CLAUDE.md`。
- 视频本质是在说明：不要靠临时 prompt 一遍遍重复交代项目背景，先把项目知识固化成“项目记忆”。

原文核心：
- `CLAUDE.md` 通常会包含构建/测试命令、关键目录、代码约定、架构决策。
- 更大的项目可以再拆到 `.claude/rules/` 中，用多个规则文件做主题化补充。
- 规则文件支持按路径条件加载，意味着不同目录可以有不同的协作约束。

实操心得：
- 在新仓库里，`/init` 应该是第一步，而不是“写了几轮 prompt 之后才补”。
- `CLAUDE.md` 适合放“项目级恒定规则”；`.claude/rules/` 适合放“局部、可演化、带作用域的规则”。
- 如果团队里有测试规范、数据库迁移规范、API 兼容策略，最好拆成独立规则文件，而不是全部塞在一个长文档里。

你可以这样理解：
- `CLAUDE.md` 是项目总说明书。
- `.claude/rules/` 是专题化操作手册。

常见误区：
- 把 `CLAUDE.md` 写成大而全的废话文档，结果真正有约束力的内容淹没了。
- 把临时任务说明写进长期记忆，导致后续每次会话都被无关约束污染。

### 2. Memory Updates：直接让 Claude 更新项目记忆

视频演示意图：
- 演示在对话中直接告诉 Claude 更新 `CLAUDE.md`，而不是手动去打开文件编辑。
- 重点是“记忆维护内嵌到工作流中”，减少打断。

原文核心：
- 早期可以用 `#` 前缀把内容追加进记忆，文章写作时该方式已变化。
- 现在直接自然语言要求 Claude 更新 `Claude.md` 即可。

实操心得：
- 这是把“经验沉淀”变成即时动作，而不是事后补文档。
- 每当你发现一个高频偏好反复出现，就应该立即沉淀，例如：
  - 永远用 `bun` 不用 `npm`
  - 测试优先跑某个子目录
  - 生成代码时必须保留某种命名风格

真正的价值：
- 减少重复指令。
- 保持多轮会话的一致性。
- 让未来的你少解释一次。

风险点：
- 不要把短期偏好写成长期规则。
- 更新记忆时最好明确写成可执行约束，而不是模糊偏好。

### 3. `@` Mentions：最快的上下文注入方式

视频演示意图：
- 演示输入 `@` 后快速选中文件、目录，或者启用/引用 MCP。
- 重点不是“有这个语法”，而是“上下文注入速度足够快，才会被高频使用”。

原文核心：
- `@文件`、`@目录`、`@mcp:server` 都可以快速加入上下文。
- Git 仓库里建议更快，并支持模糊匹配。

实操心得：
- 当任务边界明确时，优先用 `@` 限定上下文，而不是让 Claude 全仓搜索。
- `@目录` 适合让 Claude 建立一段子系统的整体认知；`@文件` 适合针对性修改。
- 如果你已经知道问题大概率在某个子模块，精准加上下文比“你自己去找”更高效，也更省 token。

最佳使用场景：
- Bug 定位时：`@相关文件 + 明确错误现象`
- 重构时：`@目录 + 说明要保持的接口边界`
- 文档补全时：`@已有实现 + 说明输出格式`

## 2. Essential Shortcuts：把常用动作压缩成肌肉记忆

### 4. `!` 前缀：立即执行 Bash

视频演示意图：
- 演示输入 `! git status` 这类命令后，终端命令直接执行，结果回流到会话上下文。
- 视频要表达的是：很多事情不需要“请求 AI 再代你请求 shell”。

原文核心：
- `!` 后接 Bash 命令，命令立刻执行。
- 无需额外模型处理，减少等待和 token 消耗。

实操心得：
- 这是 Claude Code 最重要的低摩擦能力之一。
- 它把“聊天”和“终端”合并成一个连续面。
- 对开发者来说，频繁的 `git status`、`ls`、`rg`、测试命令、日志查看，都属于应该瞬发的动作。

为什么它重要：
- 减少上下文切换。
- 降低确认成本。
- 让你保留终端思维，而不是被迫改成聊天思维。

注意事项：
- 这是实命令，不是“模拟执行”。
- 对会改状态的命令仍然要有边界意识。

### 5. 双击 `Esc`：回滚到干净检查点

视频演示意图：
- 演示先尝试一个方向，然后用双 `Esc` 回退。
- 核心感受是：你可以放心探索，因为“试错成本下降了”。

原文核心：
- 可以回滚对话、代码改动，或两者一起。
- 但 Bash 命令无法被撤销。

实操心得：
- 这是探索式开发的安全阀。
- 当你想试一个不确定方案时，不必先复制一套临时分支思路到脑子里，直接尝试，效果不好就回退。
- 它尤其适合架构试探、文案试写、重构路径比较。

真正改变的是行为模式：
- 没有回滚时，人会趋向保守。
- 有可靠回滚时，人更愿意让 Claude 先给出一个可检验方案。

边界：
- 文件改动可回退，不代表外部副作用可回退。
- 任何 shell 里对数据库、服务、远端资源的操作都不在“后悔药”范围内。

### 6. `Ctrl + R`：反向搜索历史 prompt

视频演示意图：
- 演示像 shell 历史搜索一样，快速找回过去用过的 prompt 或 slash command。
- 本质是在说：优秀 prompt 应该被复用，而不是每次从头打。

原文核心：
- `Ctrl+R` 开始搜索，再按一次继续匹配。
- `Enter` 直接运行，`Tab` 先编辑再运行。

实操心得：
- 对高频工作最有价值，例如：
  - 统一代码审查 prompt
  - 固定的排查模板
  - 常用发布前检查语句
- 如果你有一套效果很好的 prompt，先别急着做成 command；在固化之前，先用历史搜索验证它是否真的常用、是否稳定。

它解决的问题：
- 减少重复输入。
- 保留你自己的工作语言。
- 让高质量 prompt 成为“可检索资产”。

### 7. Prompt Stashing：暂存草稿 prompt

视频演示意图：
- 演示输入一半的 prompt 先存起来，去处理别的事，回来后继续。
- 重点是保护“未完成思路”。

原文核心：
- `Ctrl+S` 保存当前草稿，之后可自动恢复。

实操心得：
- 这个功能看似小，但非常贴近真实开发场景。
- 你常常在写一个复杂指令时，突然需要先查日志、跑测试、补充上下文。
- 如果没有暂存，思路很容易丢；有暂存，复杂 prompt 可以像代码草稿一样被中断和恢复。

适合的场景：
- 复杂重构任务说明
- 多约束代码生成指令
- 长篇分析型 prompt

### 8. Prompt Suggestions：让 Claude 补全你的下一步

视频演示意图：
- 演示任务结束后出现灰色建议 prompt，可用 `Tab` 或 `Enter` 接受。
- 视频表达的是“工作流补全”，不是单纯文本补全。

原文核心：
- `Tab` 接受后可编辑。
- `Enter` 直接执行。
- 可在 `/config` 中开关。

实操心得：
- 当 Claude 对当前任务链理解较好时，这个功能会降低“下一步该怎么说”的组织成本。
- 它适合线性、连续的任务，不适合高分支、强决策任务。

正确用法：
- 把它当“草案”，不是“自动驾驶”。
- 快速浏览建议是否符合你的真实意图，再决定直接执行还是编辑。

不建议盲信的场景：
- 涉及生产发布
- 涉及大范围重构
- 涉及权限敏感操作

## 3. Session Management：把 Claude Code 当持续会话环境使用

### 9. `claude --continue`：从上次会话继续

视频演示意图：
- 展示会话中断后，通过命令立刻接上原上下文。
- 视频强调“工作不会因为终端关闭而归零”。

原文核心：
- 能恢复最近一次会话。
- `cleanupPeriodDays` 可控制保留时长，默认 30 天。

实操心得：
- 这让 Claude Code 更像 IDE，而不是一次性聊天。
- 对长链路任务特别重要：迁移、排障、分阶段重构、跨天开发。
- 如果你经常做长周期任务，应该有意识地依赖“延续会话”而不是每次重新解释背景。

管理建议：
- 长期项目可以延长会话保留。
- 高保密或高变更项目，则要考虑缩短保留窗口。

### 10. Named Sessions：给会话命名

视频演示意图：
- 演示在 `/resume` 界面中区分、预览、重命名会话。
- 重点是让会话变得可管理，而不是一团无名历史。

原文核心：
- `/resume` 支持按分叉组织会话。
- 有 `P` 预览、`R` 重命名等快捷操作。

实操心得：
- 命名的价值在于“可恢复”和“可定位”。
- 当你并行推进多个任务时，不命名会话很快就会失去可读性。
- 命名应围绕任务，而不是围绕日期，例如：
  - `auth-token-refresh-bug`
  - `billing-csv-export-refactor`
  - `release-checklist-draft`

### 11. Claude Code Remote：网页开始，终端接力

视频演示意图：
- 演示在 Web 端启动任务，再拉回本地终端继续。
- 强调的是“同一任务可跨设备迁移”。

原文核心：
- Web 端或移动端发起的会话可以在本地恢复。
- 也适用于 Desktop 与移动端。

实操心得：
- 这不是“移动端写代码”，而是“随时把想法启动起来，回到主战场再深入执行”。
- 典型场景：
  - 通勤时先让 Claude 帮你列排查思路
  - 离开工位时先做方案草案
  - 回到电脑后接上本地仓库与终端工具继续

它提升的是：
- 碎片时间利用率
- 任务连续性
- 灵感到执行的转化速度

### 12. `/export`：导出完整过程留档

视频演示意图：
- 演示把整个对话导出为 Markdown。
- 重点是“过程可追溯”，而不只是得到结论。

原文核心：
- 导出内容包括你的 prompt、Claude 的响应、工具调用和输出。

实操心得：
- 这非常适合：
  - 留存复杂排障记录
  - 形成团队内部案例
  - 审计某次自动化操作
  - 回顾为什么某条路径被否定

真正价值：
- AI 协作过程从“黑箱对话”变成“可审阅工单”。

## 4. Productivity Features：降低编辑与上下文摩擦

### 13. `/vim`：把 prompt 编辑变成熟悉的文本操作

视频演示意图：
- 演示在输入框中使用 Vim 式动作编辑 prompt。
- 视频在传达：如果 prompt 是高频生产资料，就值得用高效率编辑方式处理。

原文核心：
- 提供熟悉的 Vim 移动、删除、变更等操作。
- 再次输入 `/vim` 可退出。

实操心得：
- 适合长期命令行用户。
- Claude Code 不是只接收一句话问题，很多时候你写的是长指令、约束列表、重构说明，编辑效率很重要。
- 如果你本来就有 Vim 肌肉记忆，启用它几乎是无成本增益。

### 14. `/statusline`：自定义底部状态栏

视频演示意图：
- 演示底部状态栏可显示分支、模型、token、上下文占比等信息。
- 核心是“把关键状态前置，不用频繁切换命令查看”。

原文核心：
- 可展示 Git 状态、当前模型、token 使用、上下文窗口比例、自定义脚本输出。

实操心得：
- 如果你把 Claude Code 当长期工作界面，状态栏非常值得配置。
- 最有价值的不是“展示很多信息”，而是“展示会影响决策的信息”。

优先推荐放的内容：
- 当前分支
- 当前模型
- 上下文占用比例
- 你最常担心的自定义信号（例如测试状态、环境名）

### 15. `/context`：查看 token 被谁吃掉了

视频演示意图：
- 演示打开 token 构成视图，分解系统提示、MCP、记忆、技能、历史等占比。
- 视频想强调：上下文爆掉不是玄学，是可诊断的。

原文核心：
- 可以看到系统 prompt、MCP 提示、`CLAUDE.md`、技能、历史对话等构成。

实操心得：
- 当 Claude 变慢、丢细节、回答发散时，第一件事不该是“抱怨模型”，而是先查上下文结构。
- 你需要把 token 当预算来管理。

它帮助你做的决策：
- 是否要缩短历史对话
- 是否该拆分规则文件
- 是否某个 MCP 或技能太重
- 是否应该开一个新会话

### 16. `/stats`：观察你的使用模式

视频演示意图：
- 演示统计面板，包括使用习惯、模型偏好、连续使用等。
- 这类视频不在讲“核心生产力”，而是在帮助你建立“使用反馈回路”。

原文核心：
- 可查看 usage patterns、favorite models、usage streaks 等。

实操心得：
- 如果你在认真把 Claude Code 当生产工具，统计数据能帮助你识别：
  - 自己最常做的是哪类任务
  - 哪种模型最常被你调用
  - 是否有一类操作特别适合进一步自动化

### 17. `/usage`：确认限额与节奏

视频演示意图：
- 演示查询当前限额或速率限制状态。
- 重点是：不要等撞墙才想起配额。

原文核心：
- 用来查看 rate limits。

实操心得：
- 在高强度使用日，特别是大上下文、长思考、长链任务时，限额预判能直接影响你怎么安排工作。
- 当你知道额度紧张时，应优先做：
  - 高价值任务
  - 需要人工判断的任务
  - 可复用产物沉淀

## 5. Thinking & Planning：控制 Claude 的思考方式

### 18. `ultrathink`：把思考预算留给复杂问题

视频演示意图：
- 演示在 prompt 中加入 `ultrathink`，触发更深思考。
- 视频重点不是“神秘咒语”，而是“你可以显式申请更高推理预算”。

原文核心：
- 可为内部推理分配更高 token。
- 文章写作时提到：历史上有多个层级，现在简化；若设置了 `MAX_THINKING_TOKENS`，则它优先生效。

实操心得：
- 这个能力应该保留给高复杂度问题，而不是日常小改动。
- 最适合：
  - 架构取舍
  - 难复现 bug 定位
  - 多方案比较
  - 高约束重构策略

错误用法：
- 连简单 CRUD 修改都滥用，浪费配额和时间。
- 把“更长思考”误当“更高正确率保证”。它提升的是推理空间，不是绝对正确性。

### 19. Plan Mode：先侦察，再动手

视频演示意图：
- 演示双击 `Shift+Tab` 进入计划模式，只读探索代码、生成方案，但不立即改文件。
- 这是整篇文章中最重要的视频之一，因为它直接定义了人和 AI 的权责边界。

原文核心：
- Plan mode 可以搜索代码、分析架构、探索依赖、拟定实施计划。
- 在你批准前不会执行编辑。
- 作者说自己 90% 的时间默认用计划模式。

实操心得：
- 对稍微复杂一点的任务，先计划再执行，几乎总是更优。
- 它把 Claude 从“先改再说”拉回“先理解再改”。
- 适用于：
  - 多文件修改
  - 重构
  - 新功能接入
  - 涉及外部依赖或迁移的任务

Plan mode 的真正价值：
- 降低误改概率
- 暴露隐含依赖
- 让你可以先审计划，再决定是否执行

最推荐的操作习惯：
- 先让 Claude 给出计划
- 人工检查范围、顺序、风险
- 针对计划反馈修正
- 确认后再切执行

### 20. Extended Thinking（API）

视频演示意图：
- 演示在 API 场景下看到思考块（thinking blocks）。
- 重点是：在 API 集成里，你不仅能用模型，还能调试它的推理过程。

原文核心：
- 使用 Claude API 时可启用 extended thinking，查看分步推理。

实操心得：
- 这更偏向开发者构建 AI 产品时的调试能力，而非终端日常功能。
- 价值在于：
  - 帮助理解模型为什么给出某个结论
  - 便于调试复杂逻辑链
  - 提高 agent 系统的可解释性

## 6. Permissions & Safety：把能力装进边界里

### 21. `/sandbox`：预先定义权限边界

视频演示意图：
- 演示先设置允许范围，再让 Claude 在边界内自由工作。
- 核心是“先设规则，再给速度”。

原文核心：
- 通过 `/sandbox` 配置权限边界。
- 新版本支持通配写法，例如允许整组 MCP 能力。

实操心得：
- 这是企业场景和严肃项目里非常关键的功能。
- 真正成熟的用法不是“每次手动审批”，而是“把安全边界配置成默认制度”。

推荐思路：
- 对读取类操作放宽
- 对写入、删除、外部网络、敏感命令收紧
- 对可信的自动化路径做精细授权

### 22. YOLO Mode：用速度换控制

视频演示意图：
- 演示使用危险开关，跳过大量确认。
- 作者明显在提醒：这不是“日常默认”，而是“有意识的加速模式”。

原文核心：
- `--dangerously-skip-permissions` 会一路放行。
- 适合隔离环境或可信操作。

实操心得：
- 它的价值是真实存在的，尤其是在重复、可预测、低风险任务中。
- 但它的代价也是真实存在的，一旦上下文理解错了，破坏速度也会同步提高。

只建议在这些场景考虑：
- 临时测试仓库
- 一次性沙箱环境
- 可随时重建的本地实验项目

不建议用于：
- 生产仓库
- 涉及秘钥/配置的环境
- 不熟悉的代码库

### 23. Hooks：用确定性脚本约束概率性 AI

视频演示意图：
- 演示在不同生命周期挂上 shell 命令，拦截或补充 Claude 的行为。
- 这是“机制级防线”，不是单次 prompt 技巧。

原文核心：
- 可在工具前后、权限请求、通知、子代理开始/结束等阶段触发。
- 可通过 `/hooks` 或 `.claude/settings.json` 配置。

实操心得：
- Hooks 是让 Claude Code 从“一个好用工具”升级为“可治理系统”的关键点。
- 你可以用它：
  - 拦截危险命令
  - 自动跑格式化/测试
  - 发系统通知
  - 记录审计日志
  - 接入外部工作流

深层价值：
- Prompt 约束是软约束。
- Hook 约束是硬约束。
- 对高风险动作，永远优先依赖硬约束。

## 7. Automation & CI/CD：把 Claude Code 用进非交互流程

### 24. Headless Mode：把 Claude 当 CLI 自动化节点

视频演示意图：
- 演示使用 `-p "prompt"` 非交互运行，直接输出到 stdout。
- 重点是：Claude Code 不只是聊天器，也可以嵌入脚本和流水线。

原文核心：
- `-p` 支持无交互运行，适合脚本和自动化。

实操心得：
- 这使它可以出现在：
  - CI 检查
  - 批处理脚本
  - 自动生成说明文档
  - 周期性分析任务

最佳实践：
- 输入边界明确
- 输出格式明确
- 尽量让任务幂等
- 最好对结果做后续程序化校验

### 25. Commands：把高质量 prompt 变成可复用命令

视频演示意图：
- 演示把 Markdown 文件变成 slash command，并支持参数。
- 视频表达的重点是“把 prompt 产品化”。

原文核心：
- 可以把 prompt 保存成命令，并接受参数。

实操心得：
- 当某个 prompt 足够稳定，就不该继续靠历史搜索，而应该升级为命令。
- 适合沉淀：
  - PR review 模板
  - 发布检查清单
  - 文档生成模板
  - 某类固定代码修改流程

命令化的意义：
- 降低团队协作成本
- 提高一致性
- 减少 prompt 漂移

## 8. Browser Integration：让 Claude 看到浏览器中的真实结果

### 26. Claude Code + Chrome：从“修代码”扩展到“验证结果”

视频演示意图：
- 演示 Claude 直接操作浏览器：打开页面、点击、读 console、看 DOM、截图。
- 这是从“只修改代码”转向“闭环验证”的关键能力。

原文核心：
- 能导航、点击、填表、读控制台错误、检查 DOM、截图。
- 作者给出的理想状态是：“修复 bug 并验证它有效”可以是一条指令。

实操心得：
- 它把很多以前需要人工补上的“验收步骤”交给了同一代理完成。
- 对前端、集成流程、表单流程、回归检查尤其有价值。

- Claude 可以基于真实运行结果继续调整策略。
- 这让修复与验证形成闭环，而不是只停在静态代码层面。

边界：
- 浏览器自动化提升了能力，也扩大了副作用范围。
- 涉及真实账号、真实支付、真实外部系统时，必须有隔离策略。

## 9. Advanced：Agents & Extensibility

### 27. Subagents：并行分工，而不是单线程硬撑

视频演示意图：

## 三、快速参考区的真正含义

文章最后列出的快捷键、命令、CLI flags，不只是“备忘清单”，而是把前文零散能力收束成 3 层操作面：

1. 键盘层：`Esc Esc`、`Ctrl+R`、`Ctrl+S`、`Shift+Tab`，负责交互速度。
2. 命令层：`/init`、`/context`、`/hooks`、`/sandbox`，负责环境控制。
3. CLI 层：`-p`、`--continue`、`--resume`、`--dangerously-skip-permissions`，负责自动化与会话管理。

这 3 层合起来，Claude Code 才像一个完整的开发操作系统，而不是单一聊天入口。

## 四、从这篇文章提炼出的高价值使用心得

### 1. 先“建模项目”，再“发任务”

不要一上来就让 Claude 改代码。先做这几件事：

- 跑 `/init`
- 整理 `CLAUDE.md`
- 把关键团队规范拆进 `.claude/rules/`
- 用 `@` 精准注入上下文

这样后续所有会话质量都会更高。

### 2. 复杂任务默认先走 Plan Mode

如果任务涉及多个文件、架构理解、潜在副作用，先让 Claude 只读探索并给计划，再决定执行。这样做通常比“直接改”更快，因为它减少返工。

### 3. 把 prompt 当资产管理

这篇文章其实在教你逐级沉淀：

- 一次性 prompt：先直接说
- 高频 prompt：用 `Ctrl+R` 复用
- 稳定 prompt：做成 command
- 组织级流程：做成 skill / plugin

这是一条非常清晰的知识产品化路径。

### 4. 把安全从“提醒”升级为“机制”

只靠 prompt 说“请小心”是不够的。真正靠谱的是：

- 用 `/sandbox` 划边界
- 用 hooks 做硬拦截
- 只在隔离环境里考虑 YOLO mode

这是把“好习惯”变成“不会轻易失效的制度”。

### 5. Claude Code 的上限，来自闭环能力

文章中最有战略意义的不是单个快捷键，而是这条链路：

- 读项目
- 形成计划
- 修改代码
- 运行命令
- 看浏览器结果
- 调用子代理
- 导出记录
- 沉淀成命令/技能/插件

一旦形成闭环，它就不只是回答问题，而是在参与完整软件交付流程。

## 五、适合直接照搬的个人工作流

如果你想把文章里的技巧真正用起来，可以采用下面这套顺序：

1. 新项目第一天：`/init`，补全 `CLAUDE.md`，拆出 `.claude/rules/`。
2. 日常进入任务：先 `@` 指定范围，复杂任务先开 Plan Mode。
3. 执行阶段：用 `!` 运行检查命令，用 `Esc Esc` 保证可试错。
4. 长任务管理：命名会话，必要时 `--continue` 或 `/resume` 接续。
5. 上下文变重：用 `/context` 看 token 构成，及时开新会话或减负。
6. 高价值高频动作：沉淀成 commands。
7. 高风险场景：先配 `/sandbox` 和 hooks，再放大自动化。
8. 前端/集成验证：结合 Chrome 能力做“修复 + 验证”闭环。
9. 成熟流程：升级成 skills 或 plugins。
10. 需要批量运行时：切到 headless mode 或用 Agent SDK 系统化。

## 六、对这篇文章的最终判断

这篇文章最值得学的，不是 31 个技巧本身，而是它背后的优先级排序：

- 最先学的是上下文与项目记忆。
- 接着学的是快捷操作与会话管理。
- 再往后是计划、权限、安全。
- 最后才是自动化、浏览器、子代理、插件、SDK 这些高阶能力。

这套顺序非常合理。因为如果你前面的“理解项目、控制范围、保障安全”没有建立起来，越高级的自动化只会越快地产生错误；反过来，如果基础打好了，后面的能力会让你的开发速度和可复用性成倍提升。

一句话总结这篇文章：

> Claude Code 的真正威力，不在于它会不会写代码，而在于你能不能把它放进一个有上下文、有边界、有反馈闭环、可沉淀复用的工程系统里。
